##### !question
A junior developer joins a new team and is added to the `engineering` group in the company's IAM system. The `engineering` group has a policy attached that grants read and write access to the staging environment. The developer does not need access to the production environment. Two months later, the developer moves to a different team and their manager forgets to remove them from the `engineering` group. Which of the following best describes the security risk this introduces?
##### !end-question

##### !options
a| There is no risk because the developer no longer actively uses the staging environment.
b| The developer retains permissions they no longer need, creating stale credentials that violate the Principle of Least Privilege and represent unnecessary risk.
c| The risk is limited because the developer only has access to the staging environment, not production.
d| The risk is eliminated because the IAM system will automatically revoke permissions when the developer changes teams.
##### !end-options

##### !answer
b|
##### !end-answer

#### !explanation 
Permissions that outlive their purpose are a common violation of the Principle of Least Privilege. Even if the developer does not actively use the access, stale permissions increase the number of principals with access to a resource, making the system harder to audit and creating unnecessary risk if the developer's account is ever compromised.
#### !end-explanation

##### !answer
a|
d|

## Case Study Review

Read the following account of a real-world security incident and then answer the question below.

In May 2024, a security research group called Rabbitude gained access to the codebase of the Rabbit, an AI-powered consumer device. During their [investigation](https://www.rabbit.tech/security-investigation-062524), they discovered that multiple API keys for third-party services had been hardcoded directly into the device's source code. These keys granted access to services including ElevenLabs for text-to-speech, Azure, Yelp, and Google Maps. The most sensitive of these keys, the ElevenLabs API key, provided full administrative access and could be used to read every response ever generated by any R1 device, alter device responses, replace device voices, and crash the backend by deleting voice models, which would have rendered all R1 devices non-functional.

Rabbitude notified Rabbit of the issue on May 16, 2024. Despite this notification, the keys remained active and unrotated for over a month. Rabbit did not take action until Rabbitude published their findings publicly. Even after Rabbit began rotating the keys, a fifth hardcoded key for SendGrid, which provided access to all emails sent from the `r1.rabbit.tech` domain, was initially overlooked and remained active. Rabbit later confirmed that it had identified additional secrets that were not properly stored in a secrets management tool and began moving them to a dedicated secrets management service as part of its remediation effort.

##### !question
A secret that has been removed from the current version of a codebase is no longer accessible to anyone with access to the repository.
##### !end-question

##### !options
a| True
b| False
##### !end-options

##### !answer
b|
##### !end-answer

#### !explanation 
Version control systems retain a complete history of every change made to a codebase. A secret that is removed in a subsequent commit still exists in the repository's commit history and can be retrieved by anyone with access to that history. Recovering from an accidental secret commit requires rewriting the repository's history, which is a disruptive process. This is why preventing secrets from being committed in the first place is more effective than removing them after the fact.
#### !end-explanation

##### !question
A team has enabled automated secret rotation for a database credential used by a web application. The rotation interval is set to 24 hours. A developer notices that the application occasionally fails to authenticate to the database after a rotation event occurs. Which of the following is the most likely cause of this failure?
##### !end-question

##### !options
a| The secrets management tool is generating weak credentials that the database rejects.
b| The application retrieves the database credential once at startup and caches it in memory, so it continues using the old credential after rotation until the application is restarted.
c| Automated rotation is incompatible with database credentials and should only be used for API keys.
d| The 24-hour rotation interval is too short and is causing the database to reject connections from the application.
##### !end-options

##### !answer
b|
##### !end-answer

#### !explanation 
For automated rotation to work reliably, applications must retrieve secrets dynamically at runtime rather than caching them at startup. An application that caches a credential at startup will continue using the old value after rotation, causing authentication failures when the old credential is invalidated. The solution is to design the application to retrieve the credential fresh from the secrets management tool each time it is needed.
#### !end-explanation

##### !question
A security audit reveals that a company's encryption keys have not been rotated in three years. During that period, several developers who had access to the keys have left the organization, and one internal system that used the keys was decommissioned without a formal offboarding process. Which of the following best describes the risk this situation presents?
##### !end-question

##### !options
a| There is no immediate risk because the keys have not been reported as compromised and the data they protect remains encrypted.
b| The risk is eliminated because the developers who left the organization no longer have access to the company's systems.
c| The risk is limited to the decommissioned system and does not affect any currently active systems that use the same keys.
d| The keys may have been exposed through the departing developers or the decommissioned system without the organization's knowledge, and without rotation they remain valid indefinitely, leaving any compromise undetected and unmitigated.

##### !end-options

##### !answer
d|
##### !end-answer

#### !explanation 
Keys that are never rotated accumulate risk over time because any exposure that occurred during their active lifetime, whether through departing personnel, decommissioned systems, or other means, remains exploitable indefinitely. An attacker who obtained a key years ago and has been using it undetected would retain that access until the key is rotated or revoked. Regular key rotation limits this window of persistent access even in the absence of a known compromise.
#### !end-explanation
